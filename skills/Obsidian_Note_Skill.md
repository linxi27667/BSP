## Core Philosophy: 双模式笔记法

---

### 模式 A：普通笔记 - 三要素法

所有普通笔记必须包含：

#### 1. 时间要素
- **创建时间**: 每条笔记必须包含时间戳
- **格式**: `YYYY-MM-DD HH:MM`

#### 2. 架构要素
- 使用 `##` 分隔不同主题
- 使用 `###` 分隔子主题
- 使用 `- [ ]` 和 `- [x]` 标记待办

#### 3. 重点要素
- 使用 `==高亮==` 标记核心概念
- 使用 `> [!NOTE]` / `> [!TIP]` 标记重要信息

---

### 模式 B：代码学习笔记 - LVGL 结构法

参照 `LVGL` 笔记格式，代码学习笔记必须包含以下结构：

#### 1. 标题 + 副标题说明
```markdown
# {技术名称} {主题} （{一句话概括核心特点}）
```
**示例**: `# LVGL 父级与子级关系 （长子如父 子级在上 父级在下）`

#### 2. 核心概念要点概述
```markdown
## 核心概念

### {概念名称}
- =={核心要点1}== - 详细说明
- =={核心要点2}== - 详细说明
- {要点3}
```

#### 3. 多章节详解结构
每个章节必须包含：

| 章节 | 必含元素 | 格式 |
|------|----------|------|
| **详细说明** | 功能描述、参数说明、使用场景 | 文字描述 |
| **代码示例** | 完整可运行的代码块 | ` ```c ` / ` ```python ` 等 |
| **API 表格** | 函数名、参数、返回值、说明 | Markdown 表格 |

**章节命名**: 使用中文数字（一、二、三...）+ 章节主题

#### 4. 完整示例展示实际应用场景
```markdown
## {N}、完整示例

### 示例{序号}：{场景描述}

```c
// 完整可运行的代码
void example_function(void)
{
    // 代码实现
}
```
```

#### 5. 附录 API 速查表
```markdown
## 附录：{技术名称} API 速查表

### {功能类别}
| 函数 | 说明 |
|------|------|
| `function_name()` | 功能简述 |
```

## Workflow Guidelines

### 普通笔记工作流

#### 创建/添加笔记时
1. **读取现有笔记**: 先读取 `C:\Users\30817\Documents\Obsidian Vault\笔记\我的学习笔记.md`
2. **追加新内容**: 在文件末尾追加新的时间戳主题
3. **更新元数据**: 更新文件头的 `updated` 时间

#### 更新笔记时
1. **定位目标主题**: 根据时间戳找到对应主题
2. **修改内容**: 在对应主题下更新内容
3. **更新时间戳**: 更新该主题的修改时间

### 代码学习笔记工作流

#### 创建新技术学习笔记时
1. **判断技术类型**: 确认是框架/库学习（如 LVGL、FreeRTOS）
2. **创建文件夹**: 在 `C:\Users\30817\Documents\Obsidian Vault\笔记\我的学习笔记\{技术名称}\`
3. **命名规范**: 
   - 文件夹名: `{技术名称}` (如 `LVGL`)
   - 文件名: `{中文数字}、{章节主题}.md` (如 `一、基础控件的学习.md`)
4. **遵循 LVGL 结构**: 每个文件包含标题+副标题、核心概念、章节详解、代码示例、API表格

#### 添加新章节时
1. **确定章节顺序**: 查看现有文件，确定新章节编号
2. **创建文件**: `{N}、{章节主题}.md`
3. **填充内容**: 按模板填充详细说明、代码示例、API表格

#### 更新章节时
1. **定位文件**: 找到对应章节文件
2. **修改内容**: 在对应小节下更新
3. **更新速查表**: 如有新API，同步更新附录
